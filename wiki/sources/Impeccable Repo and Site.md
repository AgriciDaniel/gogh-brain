## How it works
Use the repo capture for current command, install, license, detector, provider, and README claims.
Use the official site capture for positioning, live mode, and install-surface claims.
Use the release capture for version, date, and rule-count changes.
Use the raw SKILL.md for command routing, setup requirements, design guidance, and command categories.
Use the missing changelog capture when explaining why release pages are the version source.
Use the repo extract for summarized command mapping, design contract details, file inventory, hooks, and live mode.
Use the claim pack for adversarially reviewed confidence labels.
Use the ecosystem pack only for reception context.
Use computertech.co for the 8,000 to 15,000 token and 10 to 15 percent overhead anecdote.
Use paddo.dev only as qualitative concurrence through the claim pack unless a later source-ledger entry is added.
Use webdeveloper.com and Composio only when the claim pack flags them as supporting snapshots.
Do not use directory descriptions as independent rule-content corroboration unless the claim pack says the specific claim is confirmed.
For counts, always write dated observations.
For versions, distinguish skill-v, cli-v, and ext-v release tracks.
For changelog, say the closest captured changelog is a website page rather than a repository CHANGELOG.md.

> [!contradiction]
> Token-cost reception is not a current v3.9.1 architecture claim: the quantified March 2026 account describes v1.x, while the captured July 2026 skill is a compiled 3.9.1 skill.

> [!gap]
> The paddo.dev qualitative token source is recorded in the claim pack, but it is not present as a separate source-ledger entry in this repo snapshot.

## Pitfalls
Do not call the source bundle market-ready evidence by itself.
Gogh has separate market-ready gates.
Do not use old detector counts from May or v3.0.3 without dates.
Do not cite the website changelog as a repository CHANGELOG.md.
Do not treat practitioner reviews as primary sources for rule content.
Do not treat a directory listing as independent corroboration when it derives from the repo.
Do not collapse Skill, CLI, and Extension versions into one version number.
Do not move the token-cost anecdote into [Impeccable Setup Token Cost](../questions/Impeccable%20Setup%20Token%20Cost.md) or [Impeccable Token Cost Measurement](../experiments/Impeccable%20Token%20Cost%20Measurement.md) in this slice.
Those future notes belong to S5h.
Do not claim the exact no-finding detector exit code from the S5c source bundle.
Do not omit retrieval dates for counts.
